系统架构的演变

单体架构

所有的业务全部写在一个项目中,部署到一台服务器上,所有的请求业务都由这台服务器处理.当业务增长到一定程度的时候，服务器的硬件会无法满足业务需求。自然而然地想到一个程序不行就部署多个喽.

缺点->

  1. 所有请求的压力都在一台服务器上
  2. 所有的业务都在同一个项目中,一旦其中某个业务崩溃了,会导致整个项目运行失败.
  3. 后期如果仅仅是某个业务中的一小块需要改动,还是需要将整个项目重新打包部署.

微服务架构

Springboot框架就是专门用来进行微服务开发的 - 微服务的”脚手架” - 快速搭建微服务的项目工程.

  1. 微服务的应用不一定是分散在多个服务器上,也可以是同一台服务器上.
  2. 微服务是架构的风格,将一个大的项目模块,拆分成若干个子的业务模块,并且每个子的业务模块都是一个独立的工程,可以进行单独的部署 - 解耦合了各个模块

分布式架构

将一个大的项目模块拆分成若干个子的业务模块,但是不同的业务模块分散部署在不同的服务器上.

分布式架构着重于部署的方式

“分布式一定是微服务的””微服务不一定是分布式的”

集群架构

集群模式是不同服务器部署同一套服务对外访问，实现服务的负载均衡。区别集群的方式是根据部署多台服务器业务是否相同

“好多人干同一件事情”

“集群是为了高可用,高性能,高并发 - 三高”

“分布式的每个节点都可以进行集群”

相同的服务部署在不同的服务器上 - 不同的服务器上部署相同的服务.

电商系统 - 用户模块,订单模块,支付模块 - “拆的过程”

比如 - 订单服务 - 部署在服务器A,支付模块部署在服务器B,用户模块部署在服务器C - “分布式部署的过程”

为了防止哪天服务器C哪天不可用,会容易发生雪崩效应.因为服务器A中的订单服务依赖于服务器C的用户服务的.

可以对分布式上的每个节点进行集群操作.可以将用户模块同时部署在服务器C,服务器B,服务器N上…

一旦进行了集群,后期可以再配合一些负载均衡的策略.比如

订单服务 - 服务器A和服务器C上都存在.

原则 - 先微服务架构,然后进行分布式部署,最后再做集群.

客户端请求 -> 网关[负载均衡] -> 服务器A/服务器C

常见的集群

  1. 分布式的每个节点[微服务]可以进行集群
  2. redis做集群
  3. mysql做集群
  4. es集群

JJWT

Java Json Web Token[令牌] - JJWT是一个提供端到端的JWT创建和验证的java库,由三部分组成.

1.Header(头部) - base64编码的json字符串

2.PayLoad(载荷) - base64编码的json字符串 - 真正的有效数据[例如登录信息]

3.Signature(签名) - 使用指定算法,通过Header和PlayLoad加盐计算的字符串

补充内容

redis+token如何防止表单重复提交

  1. 在加载表单页面的controller层中生成token令牌并存储到redis以及reques作用域中.
  2. 转发到jsp页面中并将token的值存储在隐藏域中.
  3. 点击提交表单的按钮时将隐藏域的token值一起提交到服务器端.
  4. 在服务器端将得到的token和redis中的token进行比较,如果值是一样的,则顺利执行,并且在执行之后将redis中的token删除.
  5. 再次发送时redis中的token已经被删除,所以在比较时会返回false,无法进行第二次提交操作.
