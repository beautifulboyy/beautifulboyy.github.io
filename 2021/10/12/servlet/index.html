Tomcat服务器

下载 - https://tomcat.apache.org/download-80.cgi

tomcat的版本需要和jdk的版本保持一致即可.

Apache 软件基金会（Apache Software Foundation）的Jakarta 项目中的一个核心项目

轻量级的服务器,免费并且开源,支持静态资源和java后端资源的部署的.

属于web容器/服务器,别称servlet容器,jsp容器

目录结构

  1. webapps - 部署后端项目的目录

  2. bin - 可执行文件

    startup.bat - 启动服务器

    shutdown.bat - 停止服务器

  3. conf - 配置文件

    server.xml - 端口号的配置

    tomcat-users.xml

  4. lib - tomcat运行依赖的jar包 - tomcat支持servlet程序以及jsp程序的执行

    servlet-api.jar

    jsp-api.jar

  5. logs - tomcat运行时记录日记

  6. temp - 临时目录 - 随时清空的

  7. work - 工作目录

配置端口号

默认的端口号是8080,需要在conf/server.xml中进行配置

Servlet程序

Servlet - server applet - 运行在服务器端的java程序

在mvc框架[本身不是一门具体的技术,是一套思想] servlet担任的角色 - C - 控制器

servlet程序实现

1.写一个类去实现javax.servlet.Servlet[I]

2.写一个类去继承javax.servlet.GenericServlet[C] - 支持通用协议的servlet - Generic[C]实现了Servlet[I]

3.写一个类去继承javax.servlet.http.HttpServlet[C] - 专门支持http协议的Servlet - HttpServlet继承了GenericServlet

Servlet配置

方式一 - xml配置

web.xml中进行配置,每个servlet的实例都是由tomcat来实例化的.

浏览器url:http://localhost:8081/servlet-j03s/day01/first

url - 统一资源定位器,服务器上的每个资源都应拥有唯一的url,组成部分

  1. 当启动tomcat服务器的时候,它会加载web.xml文件 - web.xml文件已经在内存中.
  2. http协议 - 无状态的协议.server是不能够识别出俩次请求来自于同一个客户端的.
  3. localhost - 服务器所在主机的ip地址.
  4. 8081 - 端口号,一个端口号对应一个服务器[软件]
  5. servlet-j03s - 项目的上下文路径[同一个服务器上的项目的上下文路径应该是唯一的]
  6. /day01/first - 资源的映射路径

精确匹配的过程

  1. /day01/first到web.xml文件中去找url-pattern是/day01/first的,如果找不到,报错一个HTTP状态404.
  2. 如果找到了,继续取出url-pattern标签的上一个兄弟标签servlet-name标签体中的值 - FirstServlet
  3. 到所有的servlet标签中去寻找是否存在servlet-name是否为FirstServlet
  4. 如果存在,继续取出servlet-class - [servlet程序对应的类的全限定名]
  5. 底层反射构造出该servlet程序的实例

方式二 - 注解配置

在类上写@WebServlet注解

Servlet生命周期

init,service[doGet,doPost],destroy方法

第一阶段 - Servlet何时被实例化

  1. 如果配置了0

    如果配置了0或者正数,意味着启动tomcat的时候,就会立即初始化该servlet实例,并且利用该实例立即去调用init方法.

    如果值一样,根据上下的配置顺序.如果值不一样.数值越小,优先级越高.

  2. 如果没有配置

    当请求第一次到达的时候,tomcat容器会初始化该servlet的实例.然后利用该实例立即去调用init方法

init方法仅仅只会执行1次.

第二阶段

来自于client端的请求永远是永远是先到达service方法.如果本servlet中没有重写service方法

那么会自动进入到父类HttpServlet中的service方法,根据请求的方式来决定调用doGet方法还是doPost方法.

再在重写之后的doGet方法或者doPost方法中来处理请求和响应.

第三阶段

当我们关闭tomcat的时候,会调用destroy方法.

get请求和post请求的区别

  1. get请求一般用于获取数据,post请求一般用于需要发送数据到后台使用
  2. get请求的参数会放在url上所以安全性和隐私性会比较差,post请求的参数会放在request.body作用域中,会比较安全
  3. post请求url长度理论上没有限制,get请求有限制
  4. get请求会被缓存,post请求不会被缓存
  5. get请求不受到刷新,回退的影响,post请求则会在网页回退或者刷新后,重新进行发送
  6. get请求会被记录在浏览器的历史记录中,post请求则不会被记录
  7. get请求可以被收藏为标签,post不能被收藏为标签
  8. get请求只能进行url编码,post请求则支持多种编码
  9. get请求通常通过url地址栏请求,post通常通过表单发送数据请求

三大作用域

Request作用域

request表示一个请求,只要发出一个请求就会创建一个request,它的作用域仅仅在当前请求中有效,如果web组件之间需要共享同一个请求中的数据,只能使用请求转发.

作用 : 用于服务器间同一请求不同页面之间的参数传递;用于表单的控件值传递

Session作用域

服务器为每个会话创建一个session对象,所以session中的数据可供当前会话中的所有Servlet共享.

作用域 : 会话从用户打开浏览器开始,到关闭浏览器结束.一次会话期间只会创建一个session对象.

session的销毁

  1. session超市
  2. 浏览器关闭后,再也访问不到和该浏览器对应的session,它会在超时之后被销毁
  3. 调用session.invalidate();方法

Application(ServletContext)

ServletContext在服务创建时启动,服务关闭时销毁.一个javaWeb应用只创建一个ServletContext对象,所有的浏览器在访问服务器时都共享一个ServletContext对象,ServletContext对象一般用于在多个浏览器间共享数据时使用.

测试

在同时符合模糊匹配和后缀匹配时采用模糊匹配的结果.

结论:精确匹配>模糊匹配>后缀匹配

HttpServletRequest

1.extends javax.servlet.ServletRequest[I]

2.所有请求的信息都被封装到了HttpServletRequest对象中

HttpServletResponse

1.extends javax.servlet.ServletRespose[I]

2.所有的响应给客户端的信息全部封装到这个对象中了.

URL和URI

  1. URI：Uniform Resource Identifier，统一资源标识符 上下文路径+映射路径[唯一的,对应唯一一个资源,标识资源的]
  2. URL：Uniform Resource Location 统一资源定位符

URI是包含URL

URI，是统一资源标识符，用来唯一的标识一个资源。而URL是统一资源定位器，它是一种具体的URI，即URL可以用来标识一个资源

Cookie

api:javax.servlet.http.Cookie[C]

  1. 可以用来存放少量的信息,每个cookie存储的量不能超过4kb

  2. cookie可以在server端创建并且发送给客户端

  3. cookie是保存在客户端的

  4. cookie的value可以唯一标识一个客户端,所以cookie通常可以用来作为session的跟踪机制

    4-1. cookie是如何管理session

    4-2. 会话跟踪的技术有哪些

  5. cookie对象拥有name和value[单个]和一些可操作的选项[比如设置cookie的过期时间]

  6. 通过HttpServletResponse.addCookie(javax.servlet.http.Cookie)方法来发送cookie到客户端

  7. 每个浏览器为每个web server最多只能保存20个cookie

  8. 每个浏览器最多支持300个cookie

  9. 客户端是通过请求头信息将cookie再次发送给server端,由server端进行解析.

    HttpServletRequest.getCookies()

Session会话跟踪机制

诞生背景:session为什么要进行管理?

原因:通过http协议发送的请求,http协议无状态的协议 - server不能够判断后续的多次请求是来自于同一个客户端[server本身不标识客户端.]

  1. 何时才会创建session空间

    1-1. 当后端代码中第一次出现req.getSession();

    1-2. 第一次访问jsp文件,只要设置成false,就不会创建session空间.

  2. 当第一次请求到达req.getSession();的时候,tomcat服务器会创建一个Session空间,并且给这个空间分配一个唯一的id

    接着创建了cookie,这个cookie中存放了这个空间的id,并且通过响应头信息将这个cookie发送到client端,并且保存在client

  3. 后续的请求会带上存储sessionid的cookie一起发送到server端.由tomcat解析cookie中的session空间的id,然后将这个client联系到这个client对应的session空间
